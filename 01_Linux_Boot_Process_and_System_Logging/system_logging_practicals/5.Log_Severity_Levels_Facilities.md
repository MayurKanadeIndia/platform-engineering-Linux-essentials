## 🎯 Analogy Summary

### The Hospital Emergency Color-Code System
```
Hospital has a color-coded alert system:
  🔴 CODE RED    → Patient dying     → EVERYONE stop everything NOW
  🟠 CODE ORANGE → Critical surgery  → Senior doctors immediately
  🟡 CODE YELLOW → Urgent but stable → Attend within 30 minutes
  🟢 CODE GREEN  → Routine checkup   → Handle when free
  🔵 CODE BLUE   → Just information  → File it, no action needed

Every alert has TWO pieces of info:
  WHO raised it?   → Which department? (ICU? Emergency? Pharmacy?)
  HOW serious?     → Which color code?

Linux logging works the same:
  WHO raised it?   → FACILITY   (kernel? auth? cron? mail?)
  HOW serious?     → SEVERITY   (emergency? error? info? debug?)
```

### The Company Departments Analogy (for Facilities)
```
Company Departments:    Linux Facilities:
───────────────────     ─────────────────
HR Department       →   auth     (login/security)
IT Department       →   kern     (kernel/hardware)
Finance Department  →   mail     (email system)
Operations          →   daemon   (background services)
Administration      →   cron     (scheduled tasks)
Custom Teams        →   local0-7 (your own applications)
```

## 💡 Core Concept

> Every log message carries TWO labels:
> FACILITY (which part of the system) + SEVERITY (how serious).
> Together they form the Priority that tells rsyslog exactly
> where to route the message.
> This is what makes log filtering surgical and precise.

## 🚨 The 8 Severity Levels — Complete Reference

### The Urgency Ladder
```
        🏠 ROOF      Level 0 — EMERGENCY  ← Most critical, rarest
           │         Level 1 — ALERT
           │         Level 2 — CRITICAL
           │         Level 3 — ERROR
           │         Level 4 — WARNING
           │         Level 5 — NOTICE
           │         Level 6 — INFO
        🏠 GROUND    Level 7 — DEBUG      ← Least critical, most verbose
```

> KEY RULE: Filtering by a severity level gets THAT level
> AND everything MORE critical above it.
> journalctl -p warning = WARNING + NOTICE + ... + EMERGENCY

---

### All 8 Levels — Detailed Reference

| Level | Name | Keyword | Meaning | Real Example | Auto-Alert? |
|---|---|---|---|---|---|
| 0 | EMERGENCY | `emerg` | System completely unusable | Kernel panic, total failure | Call everyone |
| 1 | ALERT | `alert` | Immediate human action needed | RAID failure, DB corruption | Call on-call |
| 2 | CRITICAL | `crit` | Critical condition in core component | Hardware failure, disk crash | Page engineer |
| 3 | ERROR | `err` | Something failed, system still runs | Service won't start, connection refused | Slack alert |
| 4 | WARNING | `warning` | Might become a problem soon | Disk at 85%, slow query | Dashboard |
| 5 | NOTICE | `notice` | Normal but significant | Service restarted, config reloaded | Log only |
| 6 | INFO | `info` | Routine status information | Service started, user logged in | Log only |
| 7 | DEBUG | `debug` | Step-by-step developer trace | Function calls, variable values | Dev only |

---

### Severity in Action — Disk Filling Up
```
Disk at 70% → Level 6 INFO      "Disk usage at 70%"
Disk at 85% → Level 5 NOTICE    "Disk usage getting high"
Disk at 95% → Level 4 WARNING   "Disk usage critical"
Disk at 99% → Level 2 CRITICAL  "Disk almost full — services may fail"
Disk at 100%→ Level 0 EMERGENCY "No disk space — system unusable"
```

### Sample Log Lines Per Level
```
EMERGENCY: kernel: Kernel panic - not syncing: Fatal exception
ALERT:     kernel: md/raid1: Disk failure on sdb1 — RAID degraded
CRITICAL:  systemd: nginx.service entered failed state
ERROR:     nginx[1234]: connect() failed (111: Connection refused)
WARNING:   mysqld: Slow query detected: 8.432 seconds
NOTICE:    sshd[1001]: Server listening on 0.0.0.0 port 22
INFO:      systemd[1]: Started MySQL Community Server
DEBUG:     myapp[123]: Entering function authenticate_user()
```

> WARNING: NEVER leave DEBUG logging on in production!
> It generates massive data and can fill your disk in hours.

## 🏷️ Facilities — Complete Reference

| Facility | Code | Generated By | Default Destination |
|---|---|---|---|
| `kern` | 0 | Linux Kernel | `/var/log/kern.log` |
| `user` | 1 | User applications | `/var/log/syslog` |

| `daemon` | 3 | System daemons/services | `/var/log/syslog` |

| `syslog` | 5 | rsyslog itself | `/var/log/syslog` |
| `cron` | 9 | Cron scheduled tasks | `/var/log/cron.log` |
| `authpriv` | 10 | Private auth messages | `/var/log/auth.log` |
| `local0` | 16 | Custom application use | Configurable |
| `local1` | 17 | Custom application use | Configurable |
| `local2` | 18 | Custom application use | Configurable |
| `local3` | 19 | Custom application use | Configurable |
| `local4` | 20 | Custom application use | Configurable |
| `local5` | 21 | Custom application use | Configurable |
| `local6` | 22 | Custom application use | Configurable |
| `local7` | 23 | Custom application use | Configurable |

> local0 through local7 are YOUR facilities — reserved for
> custom applications, scripts, and services in real companies.

## ⚙️ Priority Calculation
```
Priority = (Facility Code x 8) + Severity Level

Example:
  auth facility  = code 4
  warning level  = level 4
  Priority       = (4 x 8) + 4 = 36
  Header shows:  <36>

rsyslog calculates this automatically.
```

## 🔀 Facility + Severity in rsyslog Rules

```bash
# Format:  facility.severity    destination

# Specific facility, all severities:
kern.*              /var/log/kern.log        # ALL kernel messages
auth.*              /var/log/auth.log        # ALL auth messages
cron.*              /var/log/cron.log        # ALL cron messages

# Specific facility, minimum severity:
mail.info           /var/log/mail.log        # mail INFO and above
auth.warning        /var/log/auth_warn.log   # auth WARNING and above
kern.err            /var/log/kernel_err.log  # kernel ERROR and above

# All facilities, one severity minimum:
*.emerg             /var/log/emergency.log   # ALL emergencies
*.warning           /var/log/warnings.log    # ALL warnings+
*.err               /var/log/errors.log      # ALL errors+

# Everything:
*.*                 /var/log/syslog          # literally everything

# Remote forwarding (enterprise):
*.*                 @192.168.1.100           # UDP to remote syslog
*.*                 @@192.168.1.100          # TCP to remote syslog (reliable)
```

## 🖥️ Commands — Severity & Facility

### journalctl Severity Filtering
```bash
journalctl -p emerg                            # emergencies only
journalctl -p err                              # errors and above
journalctl -p warning                          # warnings and above
journalctl -b -p err                           # errors this boot
journalctl -u nginx -p err                     # nginx errors only
journalctl --since "1 hour ago" -p warning     # last hour warnings+
journalctl -b -1 -p crit                       # critical from last boot
```

### logger — Generate Logs at Specific Severity & Facility
```bash
# Syntax: logger -p facility.severity "message"
logger -p user.emerg   "EMERGENCY: System is completely broken!"
logger -p user.alert   "ALERT: Immediate action required!"
logger -p user.crit    "CRITICAL: Core component failed!"
logger -p user.err     "ERROR: Something failed"
logger -p user.warning "WARNING: Something might go wrong"
logger -p user.notice  "NOTICE: Something significant happened"
logger -p user.info    "INFO: Everything is working fine"
logger -p user.debug   "DEBUG: Step by step trace here"

# Using specific facilities:
logger -p auth.warning  "Test auth warning"
logger -p kern.err      "Test kernel error"
logger -p cron.info     "Test cron info"
logger -p local0.info   "MyApp: User login successful"

# Verify entries:
sudo grep "Test auth warning" /var/log/auth.log
sudo grep "MyApp" /var/log/syslog
journalctl -p warning --since "2 minutes ago"
```

## 🎯 Choosing Severity for YOUR App Logs

```
Ask yourself:                               Use this:
────────────────────────────────────────────────────────
Does the system need to stop entirely?  →   EMERGENCY (0)
Does a human need to act RIGHT NOW?     →   ALERT (1)
Did a core component completely fail?   →   CRITICAL (2)
Did something fail but we continue?     →   ERROR (3)
Could this become a problem later?      →   WARNING (4)
Did something notable but OK happen?    →   NOTICE (5)
Is this routine status information?     →   INFO (6)
Is this only useful when debugging?     →   DEBUG (7)
```

## ✅ Key Takeaways

```
✅ Every log has TWO labels: Facility (WHO) + Severity (HOW BAD)
✅ 8 severity levels: EMERGENCY(0) to DEBUG(7) — lower = more critical
✅ Filtering by severity gets THAT level AND everything above it
✅ local0 to local7 are reserved for YOUR custom applications
✅ Priority = (Facility x 8) + Severity — auto-calculated by rsyslog
✅ rsyslog routing rules combine facility.severity to route messages
✅ Warnings are early signals — fix them before they become errors
✅ Never leave DEBUG on in production — fills disk, slows system
✅ Severity levels drive automated alerting pipelines in production
✅ Choose severity carefully when writing your own application logs
```
