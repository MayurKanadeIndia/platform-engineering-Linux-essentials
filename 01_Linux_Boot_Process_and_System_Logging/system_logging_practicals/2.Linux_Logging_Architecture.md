## 🎯 Analogy Summary

### The Hospital Communication System
```
Doctors, Nurses, Patients  =  Kernel, Apps, Services (generate events)
Hospital Intercom           =  /dev/log socket (carries log messages)
Central Reception Desk      =  rsyslog / journald (receives & sorts)
Department Filing System    =  /var/log/ files (stores sorted logs)

A doctor says: "Patient in Room 5 needs medicine"
→ Intercom carries it
→ Reception receives, categorizes it
→ Filed under "Room 5 - Medicine Request"

An app says: "Failed login attempt from 45.33.32.156"
→ /dev/log socket carries it
→ rsyslog/journald receives, categorizes it
→ Written to /var/log/auth.log
```

---

## 💡 Core Concept

> Linux logging is a **pipeline** — events flow from producers
> through a logging layer and land in the right destination files.
> Two systems work together: **journald** (captures) + **rsyslog** (routes).

## 📋 Syslog Standard

### What is Syslog?
```
Syslog is NOT a program — it is a STANDARD (a set of rules) that defines:
  • How log messages should be FORMATTED
  • How they should be CATEGORIZED
  • Where they should be SENT

Like the rules of writing a letter:
  Rules say:   Put date on top, address on left, sign at bottom
  Syslog says: Put timestamp first, then hostname, then message
```

### Syslog Message Format
```
<Priority>  Timestamp   Hostname   Process[PID]:   Message
──────────  ─────────   ────────   ─────────────   ───────
  <34>      Mar 07      myserver   sshd[1234]:     Failed password for root
```

## 🏷️ Syslog Facility Codes — WHO Generated the Log?

| Facility | Code | Source |
|---|---|---|
| kern | 0 | Linux Kernel |
| user | 1 | User-level programs |
| mail | 2 | Email system |
| daemon | 3 | System background services |
| auth | 4 | Login & security events |
| syslog | 5 | Syslog internal messages |
| cron | 9 | Scheduled tasks (cron) |
| local0–7 | 16–23 | Reserved for custom apps |

## 🚨 Syslog Severity Levels — HOW Serious Is It?

| Level | Name | Emoji | Meaning | Real Example |
|---|---|---|---|---|
| 0 | EMERGENCY | 💀 | System is unusable | Complete system failure |
| 1 | ALERT | 🚨 | Immediate action needed | Database corruption |
| 2 | CRITICAL | 🔴 | Critical conditions | Hardware failure |
| 3 | ERROR | ❌ | Error conditions | Service failed to start |
| 4 | WARNING | ⚠️ | Warning conditions | Disk at 90% full |
| 5 | NOTICE | 📋 | Normal but significant | Config file changed |
| 6 | INFO | ℹ️ | Informational message | Service started OK |
| 7 | DEBUG | 🔍 | Detailed debug trace | Step-by-step execution |

### Severity in Action — Disk Filling Up
```
Disk at 70% → Level 6 INFO      "Disk usage at 70%"
Disk at 85% → Level 5 NOTICE    "Disk usage getting high"
Disk at 95% → Level 4 WARNING   "Disk usage critical"
Disk at 99% → Level 2 CRITICAL  "Disk almost full — services may fail"
Disk at 100%→ Level 0 EMERGENCY "No disk space — system unusable"
```

## ⚙️ rsyslog — The Traditional Log Manager

### What it does
```
rsyslog = the receptionist in the hospital analogy
  • Receives ALL incoming log messages
  • Reads the facility (who) + severity (how bad)
  • Routes each message to the correct file
  • Can also forward to remote log servers
```

### rsyslog Config File
```bash
/etc/rsyslog.conf          ← main config (routing rules)
/etc/rsyslog.d/            ← additional rule files
```

### How Routing Rules Work
```bash
# Format:   facility.severity    destination
# Examples:
kern.*                    /var/log/kern.log    # ALL kernel msgs
auth.*                    /var/log/auth.log    # ALL auth msgs
*.emerg                   /var/log/emergency.log  # ALL emergencies
mail.info                 /var/log/mail.log    # mail INFO+
*.*                       /var/log/syslog      # EVERYTHING

# * = wildcard (means "all")
# One message can match MULTIPLE rules → written to MULTIPLE files
```

## 🆕 journald — The Modern Log Manager

### What it does
```
journald = modern systemd logging system (since 2011)
  • Captures EVERYTHING — kernel, services, stdout/stderr of all apps
  • Stores in BINARY format (indexed, fast, tamper-resistant)
  • Works alongside rsyslog — they complement each other
```

### rsyslog vs journald Comparison

| Feature | rsyslog | journald |
|---|---|---|
| Format | Plain TEXT | Binary (indexed) |
| Read with | cat, grep, tail | `journalctl` |
| Around since | 1980s | 2011 |
| Search speed | Slow (grep scan) | Fast (indexed) |
| Captures | Syslog sources | EVERYTHING incl. stdout |
| Tamper resistant | No | Yes (sealed) |
| Works together | ✅ Yes | ✅ Yes |

### How They Work Together
```
All events
    │
    ▼
journald (captures first, stores binary)
    │
    │ forwards to
    ▼
rsyslog (writes to /var/log/ text files)
    │
    │ can also forward to
    ▼
Remote log servers / Cloud logging
```

## 🔄 Log Rotation — logrotate

### The Problem
```
Without log rotation:
  Week 1:  syslog =   10 MB
  Month 1: syslog =   40 MB
  Month 6: syslog =  240 MB
  Year 1:  syslog = 3 GB  ← disk full → system crashes!
```

### The Solution
```
logrotate runs daily (via cron) and:
  📦 Compresses old logs (saves space)
  🔄 Renames old logs (syslog → syslog.1 → syslog.2.gz)
  🗑️  Deletes logs older than X days
  📄 Creates fresh empty log file
```

### Before vs After Rotation
```
BEFORE:                       AFTER:
syslog (2GB, current)  →     syslog        (new, empty)
                             syslog.1       (yesterday, plain)
                             syslog.2.gz    (2 days ago, compressed)
                             syslog.3.gz    (3 days ago, compressed)
                             [syslog.7.gz   deleted — too old]
```

### logrotate Config Files
```bash
/etc/logrotate.conf          ← main config
/etc/logrotate.d/            ← per-application configs
/etc/logrotate.d/nginx       ← nginx-specific rotation
/etc/logrotate.d/mysql       ← mysql-specific rotation
```

### Example Config
```bash
/var/log/syslog {
    daily               # rotate every day
    rotate 7            # keep 7 days of history
    compress            # gzip old files
    delaycompress       # compress after 1 day (not immediately)
    missingok           # don't error if file is missing
    notifempty          # don't rotate if file is empty
    postrotate
        /usr/lib/rsyslog/rsyslog-rotate   # restart rsyslog after rotate
    endscript
}
```

## 🚶 Journey of a Single Log Message

```
You type wrong SSH password
         │
         ▼
sshd detects failure → creates message:
  "Failed password for root from 1.2.3.4"
  Facility: auth (4), Severity: warning (4)
         │
         ▼
Sent to /dev/log (Unix socket)
         │
         ▼
journald receives it FIRST
  → Stores in binary journal
  → Adds metadata (boot ID, PID, timestamp)
         │
         ▼
journald forwards to rsyslog
         │
         ▼
rsyslog checks routing rules:
  auth.* → /var/log/auth.log  ✅ WRITE
  *.warn → /var/log/syslog    ✅ WRITE
  *.emerg → terminals         ❌ skip (not emergency)
         │
         ▼
Written to disk (milliseconds after the event!)
         │
         ▼
logrotate runs at midnight → manages old files
```

## 🖥️ Commands Introduced in Chapter 2

```bash
# Check if rsyslog is running
systemctl status rsyslog

# Check if journald is running
systemctl status systemd-journald

# View recent logs via journald
journalctl -b --no-pager | tail -20

# View rsyslog routing rules (without comments/blanks)
cat /etc/rsyslog.conf | grep -v "^#" | grep -v "^$"

# View logrotate main config
cat /etc/logrotate.conf

# View all log files in /var/log/
ls -lh /var/log/

# Check how long system took to boot
systemd-analyze

# See which services are slowest to start
systemd-analyze blame | head -10
```

## ✅ Key Takeaways

```
✅ Syslog is a STANDARD (set of rules), NOT a program
✅ Every log has Facility (WHO generated it) + Severity (HOW bad)
✅ 8 severity levels: EMERGENCY (0) → DEBUG (7)
✅ rsyslog = traditional router — reads rules, writes to files
✅ journald = modern capturer — gets everything first in binary
✅ journald and rsyslog WORK TOGETHER, not against each other
✅ logrotate prevents logs from filling your disk
✅ A single log event travels the full pipeline in milliseconds
✅ One event can be written to MULTIPLE files (matches multiple rules)
```
